## Errors (semantic / logical)

### 1. `:supports` has the wrong PROV-O direction (line ~120)

```
:supports  domain:Evidence, range:EnvironmentalClaim, subPropertyOf prov:wasDerivedFrom
```

`prov:wasDerivedFrom(e, e2)` means "entity e was derived from entity e2". With domain `Evidence` and range `Claim`, this reads "Evidence was derived from Claim" — backwards. The property direction is inverted relative to its PROV-O parent.

**Fix:** Either remove `:supports` entirely (keep only `:based_on` which has the correct direction), or rename it to `:supported_by` and make it the inverse of `:based_on`.

### 2. `:based_on` declares `owl:inverseOf :supports` while also being `subPropertyOf prov:wasDerivedFrom` (line ~128)

`:based_on` has the correct direction (Claim → Evidence = "Claim derived from Evidence"), but declaring it as the inverse of `:supports` (which itself is backwards) creates a logical contradiction. If `:supports` is removed or fixed, this inverse declaration breaks.

**Fix:** Remove `owl:inverseOf :supports`. If you need the inverse, define it explicitly as `owl:inverseOf :based_on` on `:supports` (once `:supports` is fixed).

### 3. `:produces` is `subPropertyOf prov:wasGeneratedBy` but has wrong direction (line ~148)

`:produces` has domain `:Execution` (an activity) and range `:Output :Evidence` (entities). But `prov:wasGeneratedBy(e, a)` means "entity e was generated by activity a" — the subject must be an entity, not an activity. This is the inverse direction.

**Fix:** Remove `subPropertyOf prov:wasGeneratedBy`. If you want PROV-O alignment, define `:generated_by` with domain `:Output :Evidence` and range `:Execution` as the subproperty instead.

### 4. `:uses` is `subPropertyOf prov:used` but has wrong domain (line ~134)

`:uses` has domain `:Method` and range `:Evidence`. `prov:used(a, e)` relates an **activity** to an entity. `:Method` is a class, not an activity — it's a `prov:Entity`. This is a type mismatch.

**Fix:** Remove `subPropertyOf prov:used`. `:uses` is a valid custom property; it just shouldn't claim to be a PROV-O subproperty.

### 5. `:applies` is `subPropertyOf prov:used` (line ~140)

`:applies` has domain `:Execution` and range `:Method`. `:Execution` is a `prov:Activity`, so the domain is correct. But `prov:used(a, e)` expects an entity as the object — `:Method` is a `prov:Entity`, so this one actually aligns. However, `:uses_input` (Execution → Evidence) also claims `subPropertyOf prov:used`. These two are both subproperties of the same PROV-O property with different ranges, which is valid but worth noting — consider whether `:applies` should instead be `subPropertyOf prov:usedPlan` or a different PROV-O property.

### 6. `:licensed_under` is `subPropertyOf schema:license` but has wrong range (line ~228)

`:licensed_under` has range `:License` (a class), but `schema:license` expects a URI (xsd:anyURI). You can't have a class-typed subproperty of a URI-typed property in OWL 2.

**Fix:** Remove `subPropertyOf schema:license`.

### 7. `:cites` is `subPropertyOf schema:citation` but has wrong range (line ~186)

`:cites` has range `owl:unionOf(:Evidence :Method :Execution :Review)` (classes), but `schema:citation` expects `xsd:string`. Same type mismatch as #6.

**Fix:** Remove `subPropertyOf schema:citation`.

### 8. `owl:imports prov:` imports the wrong namespace (line ~24)

`prov:` resolves to `http://www.w3.org/ns/prov#` (the base PROV namespace), not `http://www.w3.org/ns/prov-o#` (PROV-O). The classes and properties you reference (`prov:Entity`, `prov:Activity`, `prov:Agent`, `prov:used`, etc.) are all in PROV-O, not the base PROV namespace.

**Fix:** Change to `owl:imports <http://www.w3.org/ns/prov-o#>`.

## Redundancies

### 9. `:based_on` and `:supports` are inverses of each other (lines ~120-132)

`:based_on` is `owl:inverseOf :supports`. Having both is fine, but since `:supports` has the wrong PROV-O alignment, you only need one. Keep `:based_on` (Claim → Evidence) and drop `:supports`.

### 10. `:uses` and `:uses_input` overlap conceptually (lines ~134-146)

`:uses` (Method → Evidence) and `:uses_input` (Execution → Evidence) are structurally identical — both express "X consumes evidence as input". The only difference is the domain. Consider whether a single property with domain `owl:unionOf(:Method :Execution)` would be cleaner, or if you need the distinction for inference.

### 11. `:has_quality_control` (object property) and `:quality_controls` (datatype property) (lines ~234, ~312)

`:has_quality_control` links Evidence to a `:QualityControl` class instance. `:quality_controls` is a string property on Evidence. These serve the same purpose with different modeling approaches.

**Fix:** Pick one. If you need structured QC data (type, result), use the object property + `:QualityControl` class. If you just need a free-text label, use the datatype property. Not both.

### 12. `:version_of`, `:supersedes`/`:is_superseded_by`, and `:version` (datatype) (lines ~210-222, ~838-844)

`:version_of` (Thing → Thing), `:supersedes` (Thing → Thing), and `:is_superseded_by` (Thing → Thing) all express version relationships between entities. The datatype property `:version` (Thing → string) captures version as a label. This is a tangled versioning model.

**Fix:** Use `:supersedes` / `:is_superseded_by` as the object properties for version ordering, and `:version` as the string label. Drop `:version_of` as redundant.

### 13. `:status` is overly generic and conflicts with specific status properties (line ~846-850)

`:status` has domain `owl:Thing` with values like "active, retracted, pending, completed". But `:reproducibility_status`, `:issue_status`, `:review_status`, and `:issue_status` already exist on specific classes. A generic status property on all things will conflict with these domain-specific ones.

**Fix:** Either remove `:status` and rely on the domain-specific status properties, or narrow its domain and clarify its semantics.

## Modeling inconsistencies (not errors, but worth fixing)

### 14. `:collection_method` and `:measurement_method` are strings, not object properties (lines ~304-310)

These describe methods but have range `xsd:string`. Since you have a `:Method` class, these should be object properties with range `:Method` for proper traceability.

### 15. `:data_sources` is a string, not an object property (line ~438)

Same issue — should link to `:Source` instances.

### 16. `:osi_approved` on `:Software` is misplaced (line ~370)

OSI approval applies to a **license**, not software. This should be a property of `:License`, not `:Software`.

### 17. `:links_to` is too broad to be useful (line ~198)

Domain and range are both `owl:Thing`. This is essentially `prov:wasInfluencedBy` with a different name. It adds no constraints and can't be meaningfully queried. If it's meant to be a "golden property" for Dimension 6, consider narrowing the domain/range to specific entity pairs.

### 18. Unused prefix declarations

`foaf`, `skos`, and `time` are declared but never used. Clean them up.

### 19. Ontology metadata: `dc:date` should be `dcterms:created` (line ~17)

`dc:date` is from Dublin Core Elements (1.1), which is deprecated in favor of Dublin Core Terms. Use `dcterms:created` and optionally `dcterms:modified`.
